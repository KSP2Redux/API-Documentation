    Class CollisionManager

    Manage PhysX interactions to ignore collisions between colliders in the same vessel.

    The primary mechanism for this is Physics.IgnoreCollision(). However, doing that for every collider combination scales O(n^2). Scalability problems become measurable around 100 parts, and can take several minutes after 900+ parts.

    It is much faster to only set up ignores for parts that are actually touching. This scales close to O(n) with the collider count. However, it's difficult to test this in advance. So we just let PhysX tell us which are touching, because that's what it's designed to do.

    • Record each collider as being "owned" by a vessel (guid)
    • Wait for PhysX to report a contact.
    • CollisionMonitor receives a Physics.ContactModifyEvent for the contact. (The callback is offthread)
    • If the contact is between to colliders owned by the same vessel GUID, void the contact. (This allows them to pass through as if already ignored)
    • Log the contact in a thread-safe structure.
    • At some later point on the main thread, read the log, set do the Physics.IgnoreCollision(), and (optionally) log again so that it can be undone.

    There are some cases that require special handling:

    • Collider setups that won't generate Physics.ContactModifyEvent() callbacks (usually because they are temporarily on the same Rigidbody), but need to be ignored anyway. It is necessary to call Physics.IgnoreCollision() in advance. ("always ignore")
    • Some ignores need to persist despite the collider being removed from the vessel. ("persist after removal"). This is usually a separable item (e.g., engine shroud) significantly overlaps other colliders (e.g., the engine it contains). If the ignore is removed too soon, the resulting contact event can lead to RUD.
